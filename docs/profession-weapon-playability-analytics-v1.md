# VerseCraft 阶段1总审计与一期方案定稿（职业 / 武器 / 可玩性 / 数据闭环）

> 目标：在**不推翻现有 UI 框架、现有 JSON 契约、现有 SSE 结构、现有主游玩链路**的硬约束下，把职业与武器从“设置页标签/状态面板”升级为可执行、可验证、可回写的数据闭环；同时修复游客与时长的核心口径，确保后台活跃与留存可信。

## 0. 审计范围与结论速览

本轮审计覆盖以下真实文件（按你指定清单）：

- 职业：`src/lib/profession/types.ts`、`registry.ts`、`engine.ts`、`trials.ts`、`benefits.ts`
- 状态与入口：`src/store/useGameStore.ts`、`src/app/play/page.tsx`
- 武器与 UI：`src/components/WeaponSlotPanel.tsx`、`src/lib/registry/weapons.ts`、`src/lib/registry/types.ts`
- 任务与世界：`src/lib/tasks/taskV2.ts`、`src/lib/tasks/taskRoleModel.ts`、`src/lib/registry/world.ts`、`src/lib/registry/npcProfiles.ts`、`src/lib/registry/majorNpcDeepCanon.ts`
- 后台/数据：`src/app/api/admin/overview/route.ts`、`src/lib/admin/service.ts`、`src/db/schema.ts`、`e2e/admin-api.spec.ts`、`package.json`

**结论**（用一句话概括四条主线的最根本问题）：

- **职业系统**：当前职业“可认证、可展示”，但**缺少进入主裁决链路的硬入口**，导致玩家感知不到它在改变自己的生存方式。
- **武器系统**：当前武器“可显示、可复制指令”，但**缺少‘对策-代价-可靠性-维护’的权威闭环**，导致玩家只把它当状态面板。
- **可玩性不足**：主循环依赖叙事质量，但**结构化决策点（可复述的选择/成本/回收）密度不够**，系统反馈常停留在“提示”而非“交易与后果”。
- **后台与数据**：游客侧有 `guestId` 与事件流，但**actor/session 边界与时长口径未统一**，跨时段同一人识别在“游客→核心活跃”层仍不稳定。

## 1) 职业系统现状、真实根因、用户为什么感知不到

### 1.1 现状（从代码出发）

职业数据模型已经具备“闭环骨架”：

- `ProfessionCertificationSpec` 明确提出：**主属性门槛 + 行为证据 + 试炼任务**三条件（`src/lib/profession/types.ts`）。
- `computeProfessionState()` 会根据任务、主威胁状态、图鉴、库存/仓库数量、装备武器稳定/污染/模组等计算行为证据，并生成 `eligibilityByProfession`（`src/lib/profession/engine.ts`）。
- `useGameStore` 在多个状态变化点（任务变化、位置变化、主威胁变化、武器变化、图鉴变化）都会重新计算 `professionState`（`src/store/useGameStore.ts`）。
- UI 入口存在：在 `src/app/play/page.tsx` 与 `UnifiedMenuModal` 中能看到“当前职业/主动技能/可认证提示”等。

职业收益目前主要以**提示/轻量结算**的方式存在：

- `benefits.ts` 写了被动/主动摘要与冷却键；`useGameStore.activateProfessionActive()` 只是立 flag + 冷却计时（`professionFlags` / `professionCooldowns`），并给出“建议做什么动作”的 tip。

### 1.2 真实根因（不是概念，是机制断点）

根因不是“没有职业表”或“没有 UI”，而是以下三类断点：

1) **裁决入口缺失：职业没进入权威裁决层**

- `computeProfessionState()` 的证据来源是“已发生/已回写的状态”，它对“下一步行动如何被裁决”没有硬绑定。
- `activateProfessionActive()` 只改变 `professionState.professionFlags`，且提示语是“建议做什么”，但没有看到一个明确的、可审计的“本回合职业主动被消费 → 产生结构化效果 → 被服务端/DM 回写”的闭环。

2) **职业认证被刻意延后触发（入口弱），且依赖‘遇到认证NPC’这一软门槛**

- `play/page.tsx` 明确：未遇到认证 NPC（`hasMetProfessionCertifier`）前不提示“可认证”；这降低了误导，但也进一步降低了存在感。
- 认证触发更多像“你现在可以去点一个认证”，而不是“你玩出来了这个身份，系统在你行动里承认它”。

3) **职业证据的“可解释性”只用于展示/调试，未用于驱动任务/交易/风险结构**

- `behaviorEvidenceKeys` 记录了命中来源，但目前没有形成“证据→可用委托/可获得折扣/可触发可见窗口”的强联动结构，导致玩家看不到职业改变世界的方式。

### 1.3 用户为什么感知不到

玩家的“感知”来源于三件事：**可选择**、**可付出**、**可回收**。

- 当前职业在 UI 上更多是“显示与提示”，缺少与主循环紧耦合的**可付出/可回收**：例如“为了职业收益我必须选 A 而不是 B，且我能解释为什么”。
- 认证条件虽然更严谨，但仍容易被玩家理解为“达标后去认证一下的资格标签”，而不是“身份带来更稳定的生存策略”。

## 2) 武器系统现状、真实根因、为什么没形成玩法闭环

### 2.1 现状（从代码出发）

武器数据结构其实已经有“闭环潜力”：

- `Weapon` 有稳定度、污染、模组槽、灌注、可维护等；并新增了 `tier`、`effectSource`（继承原道具主效果/门槛）、`provenance`（来源追踪）、`equipTimeCostTurns`（换装耗时）等扩展字段（`src/lib/registry/types.ts`）。
- `src/lib/registry/weapons.ts` 明确写了兼容层策略：保留旧 `WEAPONS` 出口，同时引入 `WEAPON_TEMPLATES` + `WEAPONIZATION_RECIPES`（武器化配方）。
- `WeaponSlotPanel.tsx` 展示稳定/污染/模组/灌注/属性要求，并提供“复制指令：卸下/更换武器”（UI 不直接改状态）。
- `useGameStore.applyWeaponUpdates()` 能接收 `weaponUpdates`，支持 `weapon` 整对象回写或 `weaponId` 目录回写，并会触发 `computeProfessionState()`（职业证据里也会读武器可靠性）。

### 2.2 真实根因：闭环缺 2 个关键环节（权威 + 经济）

1) **“武器对策”没有权威裁决闭环**

- UI 目前的动作只是复制自然语言指令；是否真的装备成功、耗费回合、对威胁造成何种“窗口变化/反制提示/代价”并没有一个稳定、可验证的结构化回写口径。
- 这会让玩家产生“武器只是叙事道具/提示词道具”的心理模型。

2) **“可靠性（稳定/污染）”缺少可量化的收益/惩罚通道**

稳定/污染已经展示，但玩家必须看到它们在关键回合的“失败概率、代价、维护必要性”上产生稳定差异，否则它就是面板装饰。

3) **“武器化→维护→交易/服务”没有被组织成循环**

`WEAPONIZATION_RECIPES` 存在，但如果锻造台/服务端裁决没有把“原料消耗、费用、折扣来源、provenance”写回，并在后续维护/校准/灌注中继续使用同一套来源追踪，那么玩家不会形成长期投入。

## 3) 严苛审查：当前项目可玩性不足的关键点（一期必须补）

> 这里不做“文案优化”，只谈结构化可玩性。

### 3.1 决策点密度不足：很多系统是“提示”，不是“交易”

任务系统已经引入了人物驱动层（`taskRoleModel.ts`）与叙事层（`grantState`、`taskNarrativeLayer`），但在主游玩体验中仍容易出现：

- 玩家有很多可见任务，但**难以判断优先级**与“谁在推进我”的结构化原因；
- 人物差异被 “可接取列表同质化” 稀释（该点在 `taskRoleModel.ts` 顶部审计摘要已写出）。

一期要把“提示”升级为“交易”：

- **交易**意味着：你获得某种窗口/豁免/折扣/信息，但必须支付可验证代价（时间、原石、污染、关系债务、承诺风险）。

### 3.2 三大主循环缺少“统一货币”与“跨系统回收”

现在有多种潜在货币：原石、好感、任务后果 flag、武器污染/稳定、职业冷却/flag、时间等。

问题是：这些货币在玩家心里没有汇总成 3 条清晰循环，导致“我做这一步到底在推进什么”不够明确。

一期必须把体验重构为三个主循环（详见第 5 章），并确保每个循环都有：

- 明确的投入（代价）
- 明确的产出（可验证回写）
- 明确的下一步（可操作的下一回合）

### 3.3 入口与节奏：职业/武器入口存在，但“进入主回合”方式弱

`WeaponSlotPanel` 的核心动作是复制指令；职业的核心动作是“去认证 + 点主动”。这些入口本身没错（符合“不推翻 UI/链路”），但必须补一个“行动-裁决-回写”的稳定桥接，使它们不再像设置页。

## 4) 后台/数据现状：游客缺口、在线时长口径、跨时段同一人识别问题

### 4.1 现状（从代码出发）

已有数据基础设施：

- `analytics_events` 追加日志（幂等键 `idempotency_key`），并有 `user_sessions`、`user_daily_activity`、`user_daily_tokens`、`admin_metrics_daily` 等 rollup（`src/db/schema.ts`）。
- 后台查询在 `src/lib/admin/service.ts`，并且**已开始修复“留存只统计注册用户”**的问题：在 `getRetentionMetrics()` 中用 `actor_key = u:<userId> / g:<sessionId>` 把游客合并进 cohort（这是正确方向）。
- Presence 心跳支持游客：`pingPresence()` 接受 `sessionId` 以 `guest_` 开头时也记在线（`src/app/actions/telemetry.ts`）。
- 前端有 `guestId`（`useGameStore` 生成并持久化在 IndexedDB storage 中），且在 `play/page.tsx` 里把它作为 `trackGameplayEvent({ sessionId: guestId })` 传给服务端。
- `api/chat` 在每次 action 完成时写 `playDurationDeltaSec` 到 analytics（每次动作固定增量），用于时长 rollup（`src/app/api/chat/route.ts`）。

### 4.2 真实缺口与混乱点

1) **游客“同一人跨时段识别”在 actor/session 层不稳定**

- 当前 `guestId` 生成方式是 `guest_${Math.random().toString(36).slice(2, 10)}`（`useGameStore.ts`），长度短、纯随机、无冲突兜底；在多设备/清缓存场景会变新用户，这是允许的，但问题是：同一设备也可能在某些边界（存储异常/重置）发生漂移。
- 更关键：系统里同时存在 `guestId`（前端）与 `sessionId`（analytics 表字段）两套语义。`trackGameplayEvent` 会把 `sessionId` 为空时回落到 `anon_session`；而 `pingPresence` 则用 `hb_${memberId}` 作为 sid。这会让“同一游客”的事件在后台以多个 sessionId 分裂。

2) **在线时长口径不统一：前端累计 vs 服务端事件增量**

- 前端有 `playTimeSeconds`（游客累计）与 `addPlayTimeSeconds()`，但它是否与服务端 `playDurationDeltaSec` 保持一致、由谁权威、何时结算，目前没有一套统一口径。
- `api/chat` 用 “每次有效行动固定 +PLAY_TIME_PER_ACTION_SEC” 写入 analytics，这对“粗粒度时长”可用，但与真实停留时长会偏离；同时 `useHeartbeat` 也会持续上报心跳事件，但心跳事件的 `playDurationDeltaSec=0`。

3) **游客数据仍容易只在‘反馈/问卷’可见，而在核心活跃层偏弱**

后台已补了部分 guest 聚合（如 `getDashboardTableData()` 通过 `feedbacks/survey_responses/analytics_events/user_sessions` 汇总 guest_id），但**核心活跃与时长**仍需一个统一的“actor_key”与“session rollup”策略，否则游客会在 DAU/留存/在线统计中时隐时现。

### 4.3 非侵入式解决方向（满足硬约束）

禁止指纹、禁止打扰用户的鉴别方式，所以我们只用：

- **稳定的本地匿名 key**（已存在：`guestId`，但需要提升稳定性与一致性）
- **轻量 server set-cookie 或 local storage / indexedDB 兜底**（不要求登录、无弹窗）
- **actor_key 统一规则**：registered user 用 `u:<id>`；游客用 `g:<guestId>`；并确保全链路 sessionId/guestId 不再混用语义。

## 5) 新的产品总结构（定稿）

### 5.1 职业身份闭环（从“标签”到“身份系统”）

**目标**：职业必须同时满足“可认证、可执行、可回写、可追责”，并在主循环中产生稳定差异。

**闭环结构**（不推翻现有链路，采用“最小新增结构化回写字段 + 现有 prompt/packet 注入”）：

1) **认证**（已有）：属性门槛 + 行为证据 + 试炼任务（保留）。
2) **身份承认**（新增落点）：每回合裁决时，系统必须把“当前职业 + 是否发动主动 + 触发证据 key”以结构化方式写入 DM change set / runSnapshot（不改变 SSE 形态，只在既有 payload 增字段）。
3) **身份收益**（从提示升级到规则）：职业影响三类裁决：
   - **窗口类**：对主威胁压制/撤离/验证/交涉/溯源等动作，提供更高概率获得“可验证下一步”的 counter hint（不是自动通关）。
   - **代价类**：在特定动作上降低一小段代价（例如精神损耗 -1 这种已在文案提到的效果），但必须有消费条件（主动 flag 或证据 key 命中）。
   - **交易类**：职业影响可接委托、服务折扣来源可用性（例如溯源师更容易拿到“维护路径建议/小额折扣倾向”的资格，但仍需支付）。
4) **回收与审计**：职业主动的消费必须在一次回合后清除，并记录到 analytics 的 payload（仅记录职业 id/是否发动/命中 key，不记录原始文本）。

**一期交付形态**：

- 仍保留单职业制（你已要求）。
- 职业不再只出现在设置页/面板，而是在“你下一步行动”里产生明确结构化差异：例如当你做压制/撤离/验证/交涉/溯源动作时，系统更明确告诉你“可控窗口/不可控原因/你付了什么代价”，并把结果写回到任务/图鉴/关系/武器状态中。

### 5.2 武器对策闭环（从“面板+指令”到“装备策略”）

**目标**：武器必须是“对策工具”，而不是“装饰物/复制指令”。

闭环结构：

1) **装备/卸下/更换**：继续走“自然语言指令→服务端/DM裁决→weaponUpdates 回写”（保持 UI 兼容）。
2) **对策生效**（新增稳定规则）：当回合动作与武器 counterTags / effectSource 匹配时，触发：
   - 结构化 counter hint（写入主威胁状态或 DM 记录）
   - 稳定/污染变化（成功/失败/强行使用都要回写污染，维护可降低污染）
3) **可靠性**：稳定/污染决定“失败形态”，失败不是无事发生，而是：
   - 你浪费回合（时间代价）
   - 或造成额外精神损耗（生存代价）
   - 或暴露位置/关系惩罚（关系代价）
4) **维护与经济**：维护不是可选装饰，必须绑定服务节点（B1 锻造台/服务），形成可重复循环：武器越常用越需要维护，维护越频繁越需要原石/关系/任务回报支持。
5) **来源追踪**：所有武器化与关键维护必须写 `provenance`，后台可审计“武器为何存在/是否满足配方与费用”。

一期不做“战斗数值化”，只做“窗口化、代价化、可靠性化”的对策系统，完全契合当前叙事冒险定位。

### 5.3 生存/关系/调查三主循环（一期结构）

#### A. 生存循环（Survival）

- **输入**：时间（回合/小时）、精神、原石、位置风险、主威胁相位
- **关键决策**：压制窗口 vs 撤离窗口 vs 硬闯
- **输出**：存活、资源、威胁状态变化、短期安全窗口
- **系统支点**：职业（守灯/巡迹） + 武器可靠性（稳定/污染） + B1 服务节点

#### B. 关系循环（Relationship）

- **输入**：承诺（promise）、债务（debt）、好感/信任/恐惧、任务授予节奏
- **关键决策**：用什么交换（情报/资源/风险/代价），以及是否背约
- **输出**：服务折扣/放行/情报、任务发放、风险回收（背约反噬）
- **系统支点**：职业（齐日角） + 任务叙事层（conversation_promise / formal_task）+ NPC 主动发放 guard

#### C. 调查循环（Investigation）

- **输入**：图鉴、线索链、主威胁 counterHintsUsed、武器 effectSource、世界实体/知识库
- **关键决策**：验证一条线索（可复述） vs 继续探索（高风险）
- **输出**：可验证结论（写回图鉴/任务）、解锁行动门槛、降低未知惩罚
- **系统支点**：职业（觅兆/溯源） + 武器化来源追踪 + 任务 narrativeTrace/sourceClueIds

### 5.4 actor/session/analytics 数据闭环（一期结构）

**目标**：不打扰用户、不做侵入式指纹的前提下，让后台在“游客→核心活跃→留存→在线时长”上做到可解释、一致、可回滚。

#### 5.4.1 统一概念（必须定义清楚）

- **actor**：统计主体。注册用户 `u:<userId>`；游客 `g:<guestKey>`。
- **sessionId**：一次连续访问（或近似连续）的会话键，用于 `user_sessions` rollup 与实时在线。
- **guestKey**：游客的稳定匿名 key，来源于客户端持久化（可同步到 cookie 但不强依赖）。
- **playDurationDeltaSec**：口径统一为“有效行动时长增量”，一期不追求真实秒表停留，但要**一致与可审计**。

#### 5.4.2 一期最小改造原则（不破坏现有表）

不改表结构，只改“写入策略与聚合策略”：

- **写入策略**：所有前端 `trackGameplayEvent` 与 `pingPresence` 必须优先使用同一个 `sessionId`（游客即 `guestKey` 或 `guestKey:windowBucket`），避免出现 `anon_session`、`hb_...` 等分裂口径。
- **聚合策略**：后台统一使用 `actor_key = u:<userId> / g:<sessionId>`（已在留存中开始这么做），并扩展到 DAU/在线/时长图表。
- **时长策略**：一期采用“每次有效行动固定增量 + 心跳仅用于在线”双轨，但必须在报表中清晰标注“有效行动时长”，避免被误解为真实停留时长。

## 6) 分阶段实施计划（Stage-1 一期增强）

> 注意：本阶段只做审计与方案定稿；真正代码改造将在阶段 2 开始执行。本节给出可落地的拆解与边界。

### Phase 1（1–2 天）：闭环对齐与埋点口径修复（不动 UI）

- **职业**：把“职业主动/证据 key 命中”作为结构化字段注入 DM/裁决回写（不改变 SSE，只扩展 payload）。
- **武器**：定义“对策生效/失败形态/污染增长/维护消耗”的最小规则表（只在裁决层生效，UI 仍展示+复制指令）。
- **数据**：统一游客 `sessionId` 的生成与传递（`trackGameplayEvent` + `pingPresence` + `api/chat`），避免 `anon_session` 分裂；后台报表 actor_key 统一扩展到更多指标。

### Phase 2（3–5 天）：职业身份闭环落地（不推翻单职业）

- **职业收益落地**：把 `professionFlags`（主动已发动）与 `behaviorEvidenceKeys`（证据命中）变成裁决输入。
- **任务联动**：让职业影响“可见任务叙事层/授予节奏/服务入口提示”，形成玩家可感知的差异（不新增新 UI 页面，只用现有任务与提示块）。
- **验收**：职业不再只是面板显示，必须能在 5 次有效行动内至少触发 2 次可解释收益回写。

### Phase 3（3–5 天）：武器对策闭环落地（可靠性 + 维护 + provenance）

- **武器对策**：在主威胁/调查/撤离等动作里，把武器 counterTags/effectSource 映射成“窗口提示 + 代价结构”。
- **可靠性**：稳定/污染进入失败形态，且失败可复述（“为什么失败、付了什么代价”）。
- **维护**：绑定 B1 服务节点与原石/关系/任务后果，形成可重复循环；维护与武器化写 provenance，后台可审计。

### Phase 4（持续迭代）：三主循环体验强度拉升（内容与系统共同推进）

- **生存**：压制/撤离/硬闯三分支必须各自拥有明确成本与回收通道。
- **关系**：承诺/债务的“可追责”必须落到任务与服务折扣上。
- **调查**：线索链必须落到可执行建议（不是只更新图鉴文本）。

## 7) 每阶段验收标准（可量化）

### 职业系统验收

- **存在感**：玩家在不打开设置页的情况下，也能从行动结果中感知职业差异（至少 2 次明确回写）。
- **严谨性**：认证必须满足 3 条件（属性/证据/试炼），且资格展示与认证动作一致。
- **闭环**：职业主动必须有“发动→消费→效果→回写→冷却”的完整链路。

### 武器系统验收

- **闭环**：装备/卸下/更换一定消耗回合且不可绕过；成功/失败都必须回写稳定/污染/提示。
- **可玩性**：至少 2 类威胁/场景存在“用对武器→获得窗口；用错/强用→付代价”的稳定模式。
- **审计**：武器化/关键维护写入 provenance，后台可追溯。

### 可玩性验收（一期）

- **三循环清晰**：新手前 10 分钟能自然经历一次“生存交易”、一次“关系交易”、一次“调查验证”。
- **结构化反馈**：关键回合输出必须包含“你做了什么/为什么有效/付了什么代价/下一步建议”四要素中至少三项。

### 数据与后台验收

- **游客进入核心活跃**：游客的 enter_main_game / first_effective_action 等事件在后台可稳定聚合到同一 actor_key。
- **时长口径一致**：后台展示“有效行动时长”与“在线（presence）”分离，且跨接口一致。
- **跨时段识别**：同一设备同一游客在 7 天窗口内 actor_key 不应频繁漂移（除非用户清理存储）。

## 8) 风险与回滚策略

### 8.1 主要风险

- **职业/武器效果过强**：一旦把“提示”变成“规则”，容易破坏叙事张力或形成最优解单一路径。
- **数据口径切换导致报表断层**：sessionId/actor_key 统一后，历史数据可能出现不可比。
- **兼容风险**：旧存档缺字段、旧武器目录与新 weaponize provenance 混用。

### 8.2 回滚策略（必须可操作）

- **功能开关**：以 “worldConsequences/flags” 或环境变量控制（不改 SSE 结构），可将职业/武器的新规则回退为“仅提示不裁决”模式。
- **双写一段时间**：数据口径统一期允许“旧 sessionId + 新 guestKey”并行写入 analytics payload，用后台查询开关切换。
- **兼容读取优先**：武器回写优先接受 `weapon` 对象，其次 `weaponId`；旧存档缺 provenance 时默认 `legacy_catalog`，不阻断游玩。
